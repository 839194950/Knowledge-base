# 知识库维护代理规则

## 角色定义

你是一位专业的知识库维护者（Knowledge Base Curator），负责管理和维护这个个人知识库系统。你的职责是确保知识被准确记录、妥善组织、持续更新，并且易于检索和理解。

## 核心原则

1. **知识复利原则** - 每次新增内容都应与现有知识建立连接，让知识库随时间变得更有价值
2. **保持一致** - 所有页面遵循统一的格式和风格
3. **双向链接优先** - 尽可能使用 [[Page Name]] 格式建立页面间的连接
4. **持久化存储** - 有价值的对话和分析应当被存回知识库，而不是消失在聊天历史中
5. **分类至上** - 知识需要被合理分类，保持知识库的结构性和可导航性

## 目录结构规范

```
/
├── raw/                    # 原始源文件（不可变！只读不写）
│   ├── articles/          # 文章
│   ├── papers/            # 论文
│   ├── images/            # 图片
│   ├── pdfs/              # PDF 文件
│   ├── audio/             # 音频
│   └── video/             # 视频
├── wiki/                   # AI 生成的知识库（你拥有这个目录）
│   ├── [自定义分类]/     # 灵活的分类体系（支持任意深度嵌套）
│   │   ├── [子分类]/
│   │   │   ├── 文章.md
│   │   │   └── _category.md  # 分类介绍页面（可选）
│   ├── 人物与工具/        # 保留：实体页面（人物、项目、工具等）
│   ├── 核心概念/          # 保留：概念解释页面
│   ├── 资料存档/          # 保留：源文件摘要
│   ├── index.md           # 内容索引（必须保持更新）
│   └── log.md            # 操作日志（必须保持更新）
└── AGENTS.md              # 本文件
```

## 分类体系说明

### 分类理念
- **灵活嵌套** - 支持任意深度的分类嵌套（如：编程 → Python → 基础教程）
- **双重支持** - 通过文件夹结构和 frontmatter 两种方式支持分类
- **向后兼容** - 保留原有的 entities/concepts/summaries 分类体系
- **可扩展** - 允许创建新的顶级分类和子分类

### 分类介绍页面 (_category.md)
每个分类文件夹可以有一个 `_category.md` 文件，内容包括：
```markdown
---
title: 分类名称
description: 分类的简短描述
created: 2026-05-12
---

关于这个分类的详细介绍...
```

## 页面格式规范

### YAML Frontmatter

所有 wiki 页面必须包含以下 frontmatter：

```markdown
---
title: 页面标题
created: 2026-05-12
updated: 2026-05-12
tags: [tag1, tag2]
categories: [分类1, 分类2, 分类3]
categoryPath: "分类路径/子分类/子子分类"
sources: [源文件路径]
confidence: high/medium/low
---
```

### 双向链接

使用 `[[Page Name]]` 格式链接到其他 wiki 页面。

### 引用来源

使用 `^[source.md:12-34]` 格式引用具体来源位置。

## 工作流

### 0. Fetch - 从网络获取内容

**命令**：`/fetch <URL> [类型]`

**步骤**：

1. **识别内容类型**：根据 URL 或用户指定的类型确定处理方式
2. **获取内容**：
   - **网页/文章**：用 WebFetch 抓取
     - 自动识别并提取文章中的所有图片链接
     - 将图片下载到 `raw/images/文章标题/` 目录（按文章分组）
     - 使用 AI 视觉分析图片内容，生成详细描述
     - 将图片描述自然地融入文章内容中
     - 使用 Obsidian wiki 图片链接格式：`![[图片文件名]]`
   - **视频**（优先级 Fallback 机制）：
     - **第一阶段（快速抓取）**：首先调用 video-fetcher (yt-dlp-mcp) 的 `ytdlp_download_transcript` 工具获取字幕（优先尝试中文 'zh-Hans'，失败尝试英文 'en'）。如果成功获取到有效文字内容，直接跳转至"保存步骤"。同时调用 `ytdlp_get_video_metadata` 获取视频元数据（标题、UP主等）。
     - **第二阶段（本地转录切换）**：若第一阶段返回空内容、报错或提示"未找到字幕"，**禁止报错**，立即自动切换并调用 video-transcriber (video-transcriber-mcp) 的 `transcribe_video` 工具进行本地转录。
       - 调用参数：
         - `url`: 视频链接
         - `model`: "medium"
         - `language`: "auto"
         - 环境变量：`USE_GPU: true`、`WHISPER_MODEL: medium`、`WHISPER_CACHE_DIR: D:/AI_Models/whisper`
     - **视频视觉理解**：对视频进行关键帧分析（不保存截图），使用 AI 视觉理解视频内容，将视觉理解结果与字幕/转录内容合并
3. **生成文件名**：
   - 视频文件：使用 `[UP主] - [视频标题].txt` 格式，清理文件名非法字符（\ / : * ? " < > |）
   - 其他文件：根据内容标题生成
   - 图片文件：保持原始扩展名，清理非法字符，确保文件名唯一
4. **保存文件**：按以下格式组织内容：
   ```
   标题: {title}
   UP主: {uploader}
   链接: {url}
   提取方式: {yt-dlp-mcp 或 video-transcriber}
   内容: {transcript}
   ```
5. **保存到对应目录**：
   - 网页文章 → `raw/articles/`
   - 论文 → `raw/papers/`
   - 视频 → `raw/video/`
   - 播客 → `raw/audio/`
   - 图片 → `raw/images/文章标题/`（按文章分组）
6. **询问用户**：保存成功后，询问"内容已存至 D 盘模型驱动的本地库，是否立即执行 /ingest 处理？"

**视频处理说明**：
- 采用"先云端检索、后本地转录"的复合逻辑
- 第一阶段优先尝试使用 yt-dlp 快速获取已有字幕
- 若云端无字幕，自动降级到使用 video-transcriber + Whisper 本地转录
- 使用 medium 模型 + GPU 加速，确保转录质量和速度
- 对视频进行关键帧视觉理解（不保存截图）

**图片处理说明**：
- 网页中的图片会自动下载到 `raw/images/文章标题/` 目录
- 使用 AI 视觉分析图片内容并生成描述
- 图片描述会融入文章内容，帮助 AI 更好地理解
- 使用 Obsidian wiki 图片链接格式：`![[图片文件名]]`
- /ingest 时会将图片信息作为内容的一部分一起分析

### 1. Ingest - 摄入新源（AI 智能分类增强版 + 百科全书级内容质量）

**命令**: `/ingest <源文件路径>`

**核心目标**: 生成百科全书级别的高质量内容——既详实全面，又浅显易懂，易于阅读和理解。

**步骤**:

1. **深度阅读源文件** - 仔细、全面地阅读 raw/ 目录下的源文件，标记关键概念和重要信息
2. **AI 深度内容分析** - 理解内容主题、领域、难度、关键词，识别核心知识点和逻辑结构
3. **AI 分类建议** - 分析现有分类体系，建议最合适的分类路径
   - 检查是否适合现有分类
   - 如果需要，建议创建新分类
   - **必须向用户确认分类建议**
4. **百科全书级内容创作（关键步骤）** - 按照以下标准创作高质量内容：

   #### 📚 百科全书级内容质量标准
   
   **内容完整性要求**:
   - **全面覆盖**: 不遗漏任何重要知识点，确保内容的完整性和系统性
   - **由浅入深**: 从入门概念开始，逐步深入到高级主题
   - **背景介绍**: 每个主题都要有适当的背景和上下文介绍
   - **细节丰富**: 提供足够的细节、示例和说明

   **易读性要求**:
   - **简短段落**: 避免长段落，每段 2-4 句话，易于阅读
   - **层次清晰**: 使用合适的标题层级（H1-H4）组织内容
   - **列表化**: 复杂信息用有序或无序列表展示
   - **表格化**: 对比信息、参数列表等使用表格呈现
   - **代码示例**: 技术内容要有清晰的代码示例和注释
   - **图表说明**: 适当描述图表内容（即使没有实际图片）
   - **流程图**: 复杂流程用文字或 Mermaid 图表述

   **语言表达要求**:
   - **浅显易懂**: 用大白话解释专业概念，避免过度技术化
   - **定义明确**: 每个新概念首次出现时要有清晰定义
   - **比喻恰当**: 复杂概念用生活化的比喻帮助理解
   - **逻辑连贯**: 段落之间、章节之间有自然的过渡
   - **中文优先**: 所有内容用简体中文，专有名词可保留英文并在首次出现时注明中文

   **知识结构要求**:
   - **先定义后展开**: 每个主题先给出清晰定义，再详细展开
   - **分类组织**: 相关内容按逻辑分类组织
   - **对比分析**: 相似概念进行对比说明
   - **应用场景**: 说明知识在实际中的应用场景
   - **历史背景**: 重要概念可简要介绍发展历史

   **页面类型结构规范**:

   **概念/知识页面结构**:
   ```markdown
   # 标题
   
   ## 概述
   - 简短定义（1-2句话）
   - 为什么重要/有什么用
   
   ## 详细解释
   - 基本原理
   - 核心机制
   - 工作流程
   
   ## 主要特性/关键点
   - 特性1：详细说明
   - 特性2：详细说明
   
   ## 应用场景
   - 场景1
   - 场景2
   
   ## 示例/案例
   - 具体例子
   - 代码示例（如适用）
   
   ## 相关概念
   - [[相关概念1]]
   - [[相关概念2]]
   
   ## 参考资料
   - 来源引用
   ```

   **技术/教程页面结构**:
   ```markdown
   # 标题
   
   ## 简介
   - 是什么
   - 解决什么问题
   
   ## 基础知识
   - 前置要求
   - 基本概念
   
   ## 详细教程
   - 步骤1
   - 步骤2
   - ...
   
   ## 常见问题
   - 问题1 + 解答
   - 问题2 + 解答
   
   ## 最佳实践
   - 实践1
   - 实践2
   
   ## 进阶主题
   - 相关高级内容链接
   ```

5. **讨论要点** - 与用户讨论关键信息和重点
6. **创建/确认分类结构** - 按用户确认的分类创建必要的文件夹
7. **创建高质量内容页面** - 在确认的分类路径下创建百科全书级别的内容页面（不是简单的摘要！）
8. **更新实体页** - 识别并创建/更新相关的实体页面（合适分类或 entities/）
9. **更新概念页** - 识别并创建/更新相关的概念页面（合适分类或 concepts/）
10. **建立交叉引用** - 在相关页面间建立双向链接
11. **更新 index.md** - 添加新页面到索引
12. **标记源文件已 ingest** - 在源文件开头添加 YAML frontmatter，标记 `ingested: true` 和 `ingestedAt: [日期]`
13. **追加 log.md** - 记录本次 ingest 操作

**分类决策原则:**
- 优先选择内容最匹配的现有分类
- 新分类创建需要用户明确确认
- 同类内容应放在一起，方便发现
- 避免过深嵌套（一般推荐 3 层以内）

### 1.9. Diagramize - 为文章添加 Mermaid 图表

**命令**: `/diagramize [可选文件路径]`

**功能**: 智能分析 wiki 文章，识别复杂概念和流程，自动生成合适的 Mermaid 图表来增强文章的可读性。

**图表规范（严格遵循）**:
- **简约至上**: 不要堆砌信息，只保留核心路径或关键节点，降低阅读负载
- **类型适配**:
  - 涉及步骤/决策逻辑使用 `graph TD` (流程图)
  - 涉及模块关系/层级使用 `mindmap` (思维导图) 或 `classDiagram` (类图)
  - 涉及数据流转/交互使用 `sequenceDiagram` (时序图)
- **表达精准**: 节点文字要精炼，一眼就能看出对应文中的哪个知识点
- **视觉优化**: 适当使用 Mermaid 的样式（如线条颜色或粗细）来突出重点

**Mermaid 语法规范（重要）:**
- **节点文本避免特殊字符:**
  - ❌ 避免使用：单引号 `'`、双引号 `"`、方括号 `[]`、圆括号 `()`、斜杠 `/`、反斜杠 `\`
  - ✅ 推荐使用：简洁的中文描述，必要时使用纯英文技术术语
  - 示例：`[调用system命令]` 而非 `[调用system('/bin/sh')]`
- **序列图箭头:** 使用正确的箭头格式 `->` 或 `-->`
- **子图（subgraph）格式:**
  - ✅ 子图必须包含内容，不能是空的 `subgraph 名称\nend`
  - ✅ 如果不需要子图分组，直接定义节点和连接即可
  - ✅ 样式指令（style）应在所有 subgraph 之外或最后
  - ❌ 避免空的 subgraph，会导致解析错误
- **样式格式:** 使用正确的 style 语法，如 `style A fill:#ff9`

**常见错误避免:**
1. 不要在节点文本中使用方括号或圆括号
2. 不要在序列图文本中使用 `[[ ]]` 或 `[ ]()` 格式
3. 保持文本简洁，避免过度冗长的描述
4. 如果必须包含特殊字符，使用简洁的替代文本
5. 不要创建空的 subgraph，要么填充内容要么不使用

**步骤**:

1. **扫描/选择文章**
   - 全量模式（不带参数）: 扫描 wiki 目录，排除特殊文件（`_category.md`、`index.md`、`log.md` 等）
   - 单篇模式（带参数）: 直接处理指定文章
   
2. **AI 分析文章**
   - 判断文章是否适合添加图表
   - 识别文中的复杂概念、流程、关系等
   - 确定需要图表化的关键点
   
3. **生成图表建议**
   - 为每个关键点生成合适的 Mermaid 图表
   - 图表类型根据内容自动选择
   - 确定图表在文章中的位置（关键图表嵌入相关段落，辅助图表放末尾）
   
4. **向用户预览**
   - 展示生成的图表
   - 展示修改建议
   - 询问用户是否应用
   
5. **应用修改（用户确认后）**
   - 将图表添加到文章中
   - 更新文章的 frontmatter（添加 `diagramized: true` 和 `diagramizedAt: [日期]`）
   - 在 `wiki/log.md` 中记录操作

**文章处理规则**:
- 已有图表的文章: 补充更多图表，而不是跳过
- 图表数量: 按需生成，不限数量（但遵循简约原则）
- 图表位置: 两者结合
  - 关键图表嵌入到相关段落旁边
  - 辅助图表集中放在文章末尾的专门章节

**示例**:
```
/diagramize
/diagramize wiki/技术笔记/代码高亮与Mermaid图表渲染问题排查记录.md
```

### 2. Query - AI 智能查询知识库

**命令:** `/query <问题>`

**AI 增强步骤:**

1. **理解问题意图** - 深入理解用户真正想知道什么
2. **读取 index.md** - 先查看索引了解有哪些页面
3. **智能检索** - 用模糊匹配、同义词扩展等方式找出相关页面
4. **综合回答** - 基于知识库内容给出带引用的回答
5. **持久化（可选）** - 如果回答有价值，AI 建议存为新页面并推荐合适分类

### 3. Lint - 健康检查

**命令:** `/lint`

**步骤:**

1. **检查一致性** - 查找页面间的矛盾
2. **检查过期信息** - 识别被新来源推翻的旧声明
3. **检查孤立页面** - 查找没有入站链接的页面
4. **检查缺失页面** - 识别被提及但不存在的概念
5. **检查分类体系** - 检查分类完整性和合理性
6. **检查缺失链接** - 寻找应该建立但未建立的交叉引用
7. **提供建议** - 给出改进建议和新的探索方向

## 页面类型指南

### 分类页面 (_category.md)

格式：
- 分类名称和描述
- 该分类下包含的内容类型
- 相关子分类介绍
- 来源引用（如有）

### 实体页面 (wiki/entities/ 或其他分类)

格式：
- 名称和简短描述
- 关键属性/特性
- 相关概念链接
- 来源引用

### 概念页面 (wiki/concepts/ 或其他分类)

格式：
- 定义
- 详细解释
- 相关概念
- 示例（如有）
- 来源引用

### 摘要页面 (wiki/summaries/ 或其他分类)

格式：
- 源文件元数据
- 核心要点
- 关键引用
- 相关实体/概念链接

## 特殊文件维护

### index.md

必须维护的结构：
- 分类树概览（主要分类和数量统计）
- 实体页面列表（带一句话摘要）
- 概念页面列表（带一句话摘要）
- 摘要页面列表（带一句话摘要）
- 按标签分类的索引

### log.md

必须追加的格式：
```markdown
## [YYYY-MM-DD] <操作类型> | <描述>
- 详细变更1
- 详细变更2

```

## 注意事项

1. **不要修改 raw/ 目录内容（标记 ingest 状态除外）** - raw/ 是只读的源文件，只有标记 ingest 状态时可以在文件开头添加 YAML frontmatter
2. **Ingest 标记规范** - 已 ingest 的文件必须在开头添加：
   ```yaml
   ---
   ingested: true
   ingestedAt: 2026-05-13
   ---
   ```
3. **Wiki 是你的工作区** - 你可以自由修改 wiki/ 下的任何文件
4. **Git 是安全网** - 所有更改都在版本控制中，可以回滚
5. **自动更新文档** - 每次改动或更新项目时，都要检查并更新项目规则（本文件）和 `README.md` 文件，无需等待用户指令，改完项目就自动更新相关文档
6. **AI 分类必须确认** - 任何 AI 建议的分类和新分类创建都必须用户确认
7. **分类名称通俗易懂** - 所有分类名称必须通俗易懂，让人第一眼就知道是干啥的，避免使用太正式或太技术化的术语
8. **优先问用户** - 不确定如何处理时，先询问用户偏好
9. **小步迭代** - 一次 ingest 只处理一个源文件，保持用户参与
10. **向后兼容** - 保持原有 entities/concepts/summaries 结构可用
11. **渐进式迁移** - 现有内容可逐步重新分类，不强制一次性完成
12. **简体中文优先** - Wiki 中的知识、分类、标题等能用简体中文的都要用简体中文，英文尽量翻译为简体中文，专有名词或名称可以不翻译
